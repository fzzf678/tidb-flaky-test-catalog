## 为什么这条线值得继续做

### 历史 case 侧

把下面 `3` 个 smell 合起来后：

- `unsorted_result_assertion`
- `missing_order_by`
- `relying_on_map_iteration_order`

可以得到一个足够大的 source set：

- `union_cases = 197`

这个规模足够值得单独开 family。

但全量 patch-first 人工复读后的结果也说明：

- 这 `197` 条里只有 `100` 条真正属于结果顺序 family
- 另外 `97` 条如果继续混进来，会直接污染 subpattern 纯度

也就是说：

- `197` 适合作为输入全集
- 但不适合作为“已经验证过的 family inventory”

### 当前 TiDB 仓库侧

当前 TiDB 仓库里，单看：

- `MustQuery(...).Check(testkit.Rows(...))`

这类弱信号，数量仍然很大。

之前的扫描已经证明：

- “没 `.Sort()`”只是弱信号
- “没有 `ORDER BY`”也只是弱信号
- “返回多行”仍然是弱信号

因此这条线要做成高精度 subpattern，必须继续往机制层拆，而不能停在 smell 层。

## 全量人工 review 后保留下来的 sibling

### A. 多行 SQL 结果做精确顺序断言，但缺少稳定顺序契约

count：`44`

机制：

- 测试对多行结果做固定行序断言
- query 本身没有稳定 order contract，或者断言侧没有做无序归一化
- patch 修法是补 `ORDER BY`、补 `.Sort()`，或修 SQL 表达内部的顺序契约

formalization 状态：

- 已 formalize

### B. `map / collection / helper` 收集结果后未排序就断言

count：`35`

机制：

- 问题发生在 helper / slice / map / callback collection 层
- patch 通过排序、`ElementsMatch`、membership 等方式取消固定顺序假设

formalization 状态：

- 已 formalize

### C. `runtime / executor` 路径本身应该保序

count：`13`

机制：

- patch 的核心不是修测试断言，而是修运行时 ordered path 的顺序语义
- 常见于 `KeepOrder`、`UNION ... ORDER BY`、`UnionScan`、partition direct read 等路径

formalization 状态：

- 已 formalize

为什么优先级高：

- case 数已经足够
- 和 `b1` / `b2` 的边界清楚
- 这是当前 family 里最值得继续补出来的“非断言层” sibling

### D. `sort / comparator / tie-breaker / repeated sort` 本身不稳定

count：`6`

机制：

- patch 不是“补排序”
- 而是修排序 helper / comparator / tie-breaker / repeated unstable sort
- 它提供了非常重要的边界：看见 `sort` 也不等于已经稳定

formalization 状态：

- 已 formalize

### E. 已有 `ORDER BY`，但排序键不唯一，`LIMIT` / 边界裁剪仍会不稳定

count：`2`

机制：

- query 已经有 `ORDER BY`
- 但排序键不足以唯一决定结果顺序
- 一旦叠加 `LIMIT` / 边界裁剪，断言仍然会漂

formalization 状态：

- 已 formalize

## post-formalization backlog

### Priority 1. 校准 `b3` 的 retrieval precision

虽然这两条已经 formalize，但全量人工复读后，边界更清楚了：

- `b1`：根本没有足够强的顺序契约
- `b5`：已经有 `ORDER BY`，但 order key 还不够唯一

后续应把这条边界写得更硬，避免把两类 case 混在一起。

## 不建议再直接 formalize 的弱信号

下面这些只适合做 candidate retrieval，不适合直接判 risk：

1. `MustQuery(...).Check(testkit.Rows(...))` 没 `.Sort()`
2. SQL 没 `ORDER BY`
3. 查询返回多行
4. patch 里出现 `sort.Slice()` / `sort.Strings()`
5. patch 里出现 `ORDER BY`

这些条件单独拿出来都会有很高误报。

## formalize 时的验证要求

后续如果继续补 subpattern，必须坚持：

1. 先看 patch / test diff
2. 先证明“相同机制重复出现”
3. 再写 JSON

不要再走这条逆序：

1. 先拿旧字段 / 旧 JSON 给 case 贴标签
2. 再倒推说“这些就是 subpattern”

那样得到的不是机制提炼，只是旧标签重分组。
