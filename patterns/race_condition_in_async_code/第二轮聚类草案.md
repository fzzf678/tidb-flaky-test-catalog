# `race_condition_in_async_code` 第二轮聚类草案

这个文件是第二轮收敛用的工作板，不是正式 subpattern JSON。

目标不是再描述“大类”，而是把 `逐例梳理台账.tsv` 中已经反复出现、边界足够清楚的**细粒度机制**收敛出来，为后续正式 JSON 起草做准备。

## 当前策略

- 只收敛**代码形态、共享对象、异步路径、缺失同步、稳定修复方式**都相对一致的机制
- 每个候选都尽量写成 agent 可执行的“确定性观察信号”
- 暂时先做一批高纯度候选，不急着覆盖全部 `502` 个 case
- 对只有 1 个正例、但代码信号非常强的机制，先保留为“单例候选”

- `goroutine 闭包复用外层错误变量`
- `goroutine 闭包捕获循环变量`
- `BindHandle 单例 sctx 并发复用`

- `WaitGroup Add/Done 顺序竞态`

- 保留这些正式子模式的正例 / 边界 case 工作记录

### 1. DDL hook callback 写共享变量/标志无锁

- 收敛状态：可优先起正式 JSON
- 确定性观察信号：
  - 测试里给 `DDL` hook / callback 赋函数
  - callback 内直接写外层共享变量、标志位、指针或 test case 对象
  - 主线程或其他 goroutine 在 callback 生命周期内并发读取这些状态
  - 修复通常是 `mutex/atomic`、改成 channel 交接、或去掉共享可变对象
- 正例 case：
  - `pr-1617`
  - `pr-1687`
  - `pr-1759`
  - `pr-2054`
  - `pr-30853`
  - `pr-37466`
  - `pr-9526`
- 边界 / 不应混入：
  - `pr-53245`
  - `pr-53887`
  - 这两条更像 `DDL callback 目标 job 过滤 / once 控制`

### 2. DDL hook callback 复用共享 context / store / txn

- 收敛状态：可优先起正式 JSON
- 确定性观察信号：
  - callback 沿用外部创建的 `context`、`store`、`txn` 或类似运行时对象
  - callback 和主线程/其他路径同时访问同一个对象，而不是每次独立创建
  - 修复通常是 callback 内自建上下文，或避免跨 callback 共享事务/上下文对象
- 正例 case：
  - `pr-1903`
  - `pr-2167`
- 边界 / 相邻：
  - 不要与“写共享变量无锁”合并
  - 不要与“复用共享 TestKit/session 执行 SQL”合并

### 3. DDL hook callback 复用共享 TestKit / session 执行 SQL

- 收敛状态：可优先起正式 JSON
- 确定性观察信号：
  - callback / goroutine 内直接复用外层 `TestKit`、`session`、`test helper`
  - 同一会话对象被 callback 和主测试线程并发用来执行 SQL / DDL
  - 修复通常是每个 goroutine 单独建 session / TestKit，或把额外 SQL 挪回主线程串行执行
- 正例 case：
  - `pr-13447`
  - `pr-1656`
  - `pr-1669`
- 边界 / 相邻：
  - 不要与“同一 session 的上一个 RecordSet 未 Close 就执行下一条 SQL”混在一起

### 4. goroutine 闭包复用外层 `err` 变量

- 收敛状态：可优先起正式 JSON
- 确定性观察信号：
  - `go func(...) { err = ... }()` 或等价写法
  - goroutine 内直接给外层 `err` 赋值
  - 主线程或其他 goroutine 会并发读这个 `err`
  - 修复通常是改成 goroutine 内局部变量 `err := ...`
- 正例 case：
  - `pr-672`
  - `pr-43704`
- 混合但相关的 case：
  - `pr-625`
  - `pr-27436`
  - 这两条还混有 `WaitGroup/lifecycle` 问题，不应直接当成纯净锚点

### 5. goroutine 闭包捕获循环变量

- 收敛状态：可优先起正式 JSON
- 确定性观察信号：
  - 在 `for range` / 普通循环里启动 goroutine
  - 闭包直接用循环变量 `i` / `store` / `sql` / `plan` / `case`，没有做重新绑定
  - 并发子测试或后台 goroutine 因此读到错误对象
  - 修复通常是把循环变量作为参数传入闭包，或在循环体内新建局部副本
- 正例 case：
  - `pr-14549`
  - `pr-23866`
  - `pr-33862`
  - `pr-36202`
- 混合但相关的 case：
  - `pr-31244`
  - 这条还混有 goroutine 未等待退出

### 6. BindHandle 单例 `sctx` 并发复用

- 收敛状态：可优先起正式 JSON
- 确定性观察信号：
  - 某个长期存活的 handle / manager 持有单例 `sctx`
  - 多条路径会并发进入这个 handle，并共用该 `sctx`
  - 修复通常是给关键区加锁，或避免把单例 session 上下文暴露给并发路径
- 正例 case：
  - `pr-49092`
  - `pr-49164`
- 边界 / 相邻：
  - 不要和 `BindHandle 共享 parser 并发使用` 合并

### 7. BindHandle 共享 parser 并发使用

- 收敛状态：单例候选，但代码信号很强
- 确定性观察信号：
  - `BindHandle` 或相似组件持有共享 parser 实例
  - 多个 goroutine / 并发更新路径直接复用这个 parser
  - 修复通常是每次新建 parser，或在共享 parser 外层加锁
- 正例 case：
  - `pr-10321`
- 备注：
  - 目前只有一个高纯度正例，但机制非常确定，可以先保留

### 8. 同一 session 的上一个 `RecordSet` 未 Close 就执行下一条 SQL

- 收敛状态：单例候选，但代码信号很强
- 确定性观察信号：
  - 同一 `session` / `TestKit` 返回的 `RecordSet` 尚未 `Close`
  - 随后立刻在同一会话上继续执行下一条 SQL
  - 前一个结果集的清理 / schema 变更 / 后台操作和后续 SQL 竞争
  - 修复通常是显式 `Close` 旧 `RecordSet`，或避免同一 session 重入
- 正例 case：
  - `pr-49348`
- 边界 / 相邻：
  - `pr-35134` 更像 `RecordSet close / lifecycle`
  - `pr-44438`
  - `pr-44783`
  - 这两条更像 `DDL callback 与主线程共享 RecordSet/jobID/err`

### 9. 测试改写全局 lease 干扰后台 refresh / loader

- 收敛状态：候选已成型，但还要继续补 case
- 确定性观察信号：
  - 测试直接改 `bindinfo.Lease` / stats lease / 同类全局 lease 开关
  - domain / loader / background worker 仍然存活
  - 断言会与后台 refresh / reload / drop / capture 路径竞争
  - 修复通常是把 lease 设为 `0` 或显式停掉后台 worker，并保证修改时机先于 bootstrap / loop 启动
- 正例 case：
  - `pr-66559`
- 混合但相关的 case：
  - `pr-7901`
  - 这条还混有 `SetSystemTZ` 一次性全局初始化问题

### 10. 多个 cop iterator 共享可变 `KV request / RequestBuilder`

- 收敛状态：单例候选，但代码信号很强
- 确定性观察信号：
  - 多个 iterator / worker 并发运行
  - 它们复用同一个 `RequestBuilder` 或可变 request 对象
  - 后续构建 / 发送请求时会互相覆盖状态
  - 修复通常是每个 iterator 独立构建 request
- 正例 case：
  - `pr-61376`

### 11. `WaitGroup.Add(1)` 晚于任务投递，导致 `Done()` 抢先发生

- 收敛状态：单例候选，但代码信号很强
- 确定性观察信号：
  - 任务先被送入 channel / 队列，worker 可以立刻执行
  - `WaitGroup.Add(1)` 却在投递之后才做
  - worker 可能先 `Done()`，把计数打成负数或造成收尾错乱
  - 修复通常是先 `Add(1)`，再投递任务
- 正例 case：
  - `pr-66703`
- 边界 / 不应混入：
  - `pr-42129` 更像 `WaitGroup Add/Wait` 的 shutdown sequencing

### 12. 并发 callback 共享结果 slice，`append/reset/snapshot` 未统一加锁

- 收敛状态：单例候选，但代码信号很强
- 确定性观察信号：
  - callback / hook 会把结果写入共享 slice
  - 主线程还会对同一个 slice 做 `reset`、读快照或断言
  - 缺少统一的 `mutex` / 所有权交接
  - 修复通常是所有 `append/reset/snapshot` 都放到同一把锁下
- 正例 case：
  - `pr-65523`

## 下一步待扩展方向

当前草案里列出的稳定候选已经全部落成正式 JSON。下一轮不再泛泛地“继续找”，而是按下面 **6 个高 ROI 方向**继续往下拆。

这 6 个方向都属于 `race_condition_in_async_code` family 内部的继续细化，不是跨 family 的通用 backlog。

### 方向 1. `manager / handle / cache` 级共享 map 并发读写

- 优先级：最高
- 为什么先做：
  - 历史 case 数量多
  - grep 信号强
  - 很容易继续拆成 agent 可执行的小粒度 subpattern
- 已落成 / 已覆盖的相关正式 subpattern：
  - `并发 callback 共享结果 slice append/reset/snapshot 未统一加锁`
  - `共享摘要收集器 Reset/Update 交错竞态`

这条线在首轮细拆后，当前最值得继续 formalize 的，不是一个大簇，而是下面 **5 个更细的候选子簇**：

#### 方向 1A. `concurrent map write`：共享运行时 map 裸读写

- 收敛状态：高优先级，适合先 formalize
- 确定性观察信号：
  - `manager / handle / mock / runtime state` 上有长期存活的 `map[...]...` 字段
  - 多个 goroutine / callback / worker / handler 会并发读写这个 map
  - patch 的核心修法是：
    - `mutex / RWMutex`
    - `sync.Map`
    - 或把共享 map 改成 worker-local / request-local
- 正例 case：
  - `pr-12868` (`DirtyDB.tables`)
  - `pr-21836` (`MPPTaskHandler.TunnelSet`)
  - `pr-22428` (`MPPTaskHandler.TunnelSet`)
  - `pr-32945` (`ForListColumnPruning.valueMap`)
  - `pr-44339` (`groups` map)
  - `pr-49354` (`RunawayManager.metricsMap`)
- 边界 / 相邻：
  - `pr-24988`
  - `pr-31714`
  - 这两条更像 “callback / stream / collector 共享 map+slice” 复合簇

这条当前还可以继续往下拆成 **2 个更细的候选子模式**：

- `长期存活的 manager / handler / mock client 持有共享 map，多个并发入口直接读写`
  - 候选正式中文名：
    - `长期存活的 manager/handler 共享 map 被多个并发入口直接读写`
  - 代码信号：
    - struct 字段就是共享 map
    - 名字通常像 `TunnelSet` / `groups` / `metricsMap`
    - 并发入口通常是 `register/get/list/activate/derive` 这一类方法
  - 初始检索锚点：
    - `TunnelSet`
    - `metricsMap`
    - `groups`
    - `registerTunnel`
    - `getAndActiveTunnel`
    - `DeriveChecker`
  - 正例 case：
    - `pr-21836`
    - `pr-22428`
    - `pr-44339`
    - `pr-49354`
  - 备注：
    - 这组最像可以直接 formalize 成独立 subpattern 的一簇

  - 准正式字段草稿：
    - `summary`
      - 长期存活的 manager、handler 或 mock client 把共享 map 挂在 struct 字段上，而 `register/get/list/activate/derive` 等多个并发入口会直接读写这个 map。
    - `mechanism`
      - 这类对象通常会跨请求、跨 callback、跨 worker 长时间存活；一旦共享 map 既承担注册又承担查询或激活逻辑，而没有 `mutex/RWMutex/sync.Map`，就很容易在单测并发路径、mock server、manager runtime 中触发 `concurrent map read and map write` 或 race detector。
    - `signals_required`
      - 共享 map 是长期存活对象的字段，而不是函数内局部变量。
      - 至少两个并发入口会访问它，例如 `register/get/list/activate/derive`。
      - 这些入口之间存在读写或写写并发，而不是严格串行。
      - 修复核心是给这个 map 增加 `mutex/RWMutex/sync.Map`，或把共享 map 改成 request-local / worker-local。
    - `signals_optional`
      - map 名字常见为 `TunnelSet`、`groups`、`metricsMap`。
      - race 出现在 mock MPP、resource manager、runaway manager 一类长期存活组件中。
      - patch 里会新增 `tunnelSetLock`、`RWMutex`、`SyncMap`、`Load/Store` 等关键词。
    - `negative_guards`
      - 如果问题只发生在单个 `List/IsXxx` 只读 helper 漏拿 `RLock`，更接近 `只读 helper/List/IsXxx 访问共享 map 却漏掉 RLock`。
      - 如果问题核心是测试直接 reset / replace 整个 cache 指针，更接近方向 `1C`。
      - 如果问题核心是把共享 map alias 给外部对象，而不是并发入口直接操作 map，更接近方向 `1E`。
    - `related_subpatterns`
      - `并发 callback 共享结果 slice append/reset/snapshot 未统一加锁`
      - `懒初始化映射或指针需要一次性发布`
    - `examples.positive`
      - `pr-21836`
      - `pr-22428`
      - `pr-44339`
      - `pr-49354`
    - `examples.boundary`
      - `pr-12868`
      - `pr-32945`

- `执行期 helper / pruning / dirty-state 结构里的懒建 map 被并发 worker 读写`
  - 候选正式中文名：
    - `执行期 helper/pruning 结构里的共享 map 被并发 worker 读写`
  - 代码信号：
    - map 是 planner / partition / dirty-state 结构内部字段
    - 由并发执行路径触发延迟填充或并发查询
    - 名字通常像 `tables` / `valueMap`
  - 初始检索锚点：
    - `valueMap`
    - `tables map`
    - `DirtyDB`
    - `LocatePartition`
    - `GetDirtyTable`
  - 正例 case：
    - `pr-12868`
    - `pr-32945`
  - 边界 / 相邻：
    - `pr-3913`
    - `pr-40754`
    - 这两条虽然也是 map，但更像 `SessionVars` / session-state 主线

#### 方向 1B. 共享 map 的读路径 / 清理路径漏锁

- 收敛状态：高优先级，适合单独成簇
- 确定性观察信号：
  - 写路径已经有锁，或大多数 helper 已经走锁
  - 但某个 `Get / List / Clear / Cleanup / IsXxx` 路径漏了同一把锁
  - 典型症状是：
    - `concurrent map read and map write`
    - cleanup 时 race
    - 只在只读 helper / reset helper 上报 race
- 正例 case：
  - `pr-40572` (`Handle.IsTableLocked` 读 `tableLocked` 漏 `RLock`)
  - `pr-42019` (`RuntimeStatsColl.rootStats` 清理路径漏锁)
  - `pr-44577` (`TimerStore.memoryStoreCore.List` 读 `id2Timers` 漏锁)
- 边界 / 相邻：
  - `pr-44446`
  - 这条不只是单一读路径漏锁，而是 `namespaces / id2Timers / watchers` 整体都需要统一互斥

这条当前也可以继续拆成 **2 个更细的候选子模式**：

- `只读 helper / IsXxx / List 路径忘记拿 RLock`
  - 候选正式中文名：
    - `只读 helper/List/IsXxx 访问共享 map 却漏掉 RLock`
  - 代码信号：
    - 写路径已经通过 `mu.Lock` 保护
    - 只读 helper 直接访问共享 map / state
    - patch 只是在读 helper 外层补 `RLock/RUnlock`
  - 初始检索锚点：
    - `IsTableLocked`
    - `List(`
    - `RLock`
    - `RUnlock`
    - `tableLocked`
    - `id2Timers`
  - 正例 case：
    - `pr-40572`
    - `pr-44577`
  - 备注：
    - 这是很适合 agent 的模式，因为函数名和修法都很稳定

  - 准正式字段草稿：
    - `summary`
      - 共享 map 或共享状态的写路径已经受 `mutex/RWMutex` 保护，但只读 helper（如 `IsXxx`、`Get`、`List`）直接访问共享字段时漏掉了 `RLock`。
    - `mechanism`
      - 这类问题最迷惑人的地方在于“整体上看起来已经有锁”，但某个只读 helper 因为实现过短、调用太频繁或被误认为天然安全，直接绕过了读锁；一旦另一个 goroutine 正在更新或删除同一 map/state，就会在 helper 这条读路径上暴露 race。
    - `signals_required`
      - 同一个共享 map/state 的写路径已经显式使用 `mu.Lock`。
      - 某个只读 helper / `IsXxx` / `Get` / `List` 直接访问该共享字段。
      - helper 本身没有 `RLock/RUnlock`，且也没有清楚的“caller already holds lock”约束。
      - 修复核心是在 helper 外层补 `RLock/RUnlock`，或显式拆出一个 no-lock 版本给持锁调用方。
    - `signals_optional`
      - 函数名常带 `Is`、`Get`、`List`。
      - 共享字段名常见为 `tableLocked`、`id2Timers` 这类 map/state。
      - patch 里可能同时引入 `alreadyLocked` / `withoutLock` 这种辅助参数，避免重复加锁。
    - `negative_guards`
      - 如果 race 是在 `clear/reset/reuse/delete` 阶段暴露，且问题在 cleanup 漏锁，更接近 `cleanup/reuse/reset 路径清空共享 map 时漏锁`。
      - 如果整个结构从一开始就没有任何同步，更接近 `concurrent map write` 主簇。
      - 如果问题是 `watchers/namespaces/id2Timers` 一整组状态都没统一加锁，而不只是只读 helper 漏锁，更接近 timer store 整体互斥子簇。
    - `related_subpatterns`
      - `cleanup/reuse/reset 路径清空共享 map 时漏锁`
      - `长期存活的 manager/handler 共享 map 被多个并发入口直接读写`
    - `examples.positive`
      - `pr-40572`
      - `pr-44577`
    - `examples.boundary`
      - `pr-44446`

- `cleanup / reuse / reset 路径清空共享 map 时漏锁`
  - 候选正式中文名：
    - `cleanup/reuse/reset 路径清空共享 map 时漏锁`
  - 代码信号：
    - 平时读写可能大体受控
    - 但在 `clear/reset/reuse/delete all` 这种路径里直接删 map entry
    - race 往往只在复用或回收阶段暴露
  - 初始检索锚点：
    - `reuse`
    - `Clear`
    - `Reset`
    - `delete(`
    - `rootStats`
  - 正例 case：
    - `pr-42019`
  - 边界 / 相邻：
    - `pr-44446`
    - `pr-44562`
    - 它们更像“整组共享状态 reset 需统一加锁”，不只是 map cleanup 单点漏锁

#### 方向 1C. 测试 / helper 直接 reset 或 replace 共享 cache / handle 指针

- 收敛状态：高优先级，agent 可操作性强
- 确定性观察信号：
  - test helper 直接 `Clear / Reset / Replace` 某个全局或 domain 级 cache / handle / pointer
  - 背景 reader / worker 还在并发使用旧值或新值
  - 修复通常是：
    - 不再直接替换
    - 改成锁内 reset
    - 或先停 reader / domain / background loop
- 正例 case：
  - `pr-2833` (测试直接替换 `RegionCache`)
  - `pr-39802` (测试里直接替换 `infoCache` / `InfoCache.cache`)
- 边界 / 相邻：
  - `pr-37766`
  - 这条更像 owner switch 期间的 progress cache 生命周期问题

#### 方向 1D. 解锁后继续使用共享 cache entry / pointer

- 收敛状态：单例候选，但代码信号很强
- 确定性观察信号：
  - 在锁内拿到共享对象指针
  - 解锁后继续返回 / 保存 / 访问这个指针指向的内部字段
  - 其他 goroutine 会在锁保护下更新同一 cache entry
  - 修复通常是：
    - 锁内复制稳定字段
    - 不把内部共享对象直接暴露到锁外
- 正例 case：
  - `pr-5171` (`RegionCache` 解锁后继续使用共享 `*Region`)
- 边界 / 相邻：
  - `pr-19198`
  - 这条更像共享索引 / cursor 字段原子化，不是“解锁后继续持有指针”

#### 方向 1E. 共享 map 直接 alias 给外部，缺少 copy-on-read

- 收敛状态：值得保留，但更靠近 `copy-on-write / publish-once` 主线
- 确定性观察信号：
  - 内部共享 map / slice 直接挂到返回对象或外部对象上
  - 外部代码以为拿到的是 snapshot，实际上仍和内部共享同一底层对象
  - 修复通常是深拷贝、copy-on-read、或不暴露内部 map
- 正例 case：
  - `pr-39721` (`authUsers` map 直接 alias 给 `BindableStatement.Users`)
- 边界 / 相邻：
  - `pr-13465`
  - `pr-45563`
  - `pr-45579`
  - 这些更适合归到方向 5 的 `copy-on-write / clone-then-swap`

#### 方向 1 当前明确不继续硬塞进主线的 case

- 更适合方向 2（共享 session / context / vars）：
  - `pr-3913`
  - `pr-40754`
- 更适合方向 3（stats / cache / collector）：
  - `pr-6901`
  - `pr-7305`
  - `pr-30550`
  - `pr-44562`
  - `pr-44748`
  - `pr-46090`
- 更适合方向 5（copy-on-write / publish-once）：
  - `pr-9008`
  - `pr-13647`
  - `pr-15233`
  - `pr-39721`
  - `pr-45563`
  - `pr-45579`

#### 方向 1 的下一步

- 先从 `1A concurrent map write` 和 `1B 读路径 / 清理路径漏锁` 里各挑一批高纯度 case

## 暂不并入本轮主簇的边界

- `cancel / close / teardown` 生命周期问题
- `timing / sleep / Eventually / failpoint` 窗口问题
- `state-machine / owner-transfer / stale-state` 问题
- `exact-order / exact-count` brittle assertion
