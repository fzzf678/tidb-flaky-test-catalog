# `race_condition_in_async_code` 第二轮聚类草案

这个文件是第二轮收敛用的工作板，不是正式 subpattern JSON。

目标不是再描述“大类”，而是把 `逐例梳理台账.tsv` 中已经反复出现、边界足够清楚的**细粒度机制**收敛出来，为后续正式 JSON 起草做准备。

## 当前策略

- 只收敛**代码形态、共享对象、异步路径、缺失同步、稳定修复方式**都相对一致的机制
- 每个候选都尽量写成 agent 可执行的“确定性观察信号”
- 暂时先做一批高纯度候选，不急着覆盖全部 `502` 个 case
- 对只有 1 个正例、但代码信号非常强的机制，先保留为“单例候选”

### 1. DDL hook callback 写共享变量/标志无锁

- 收敛状态：可优先起正式 JSON
- 确定性观察信号：
  - 测试里给 `DDL` hook / callback 赋函数
  - callback 内直接写外层共享变量、标志位、指针或 test case 对象
  - 主线程或其他 goroutine 在 callback 生命周期内并发读取这些状态
  - 修复通常是 `mutex/atomic`、改成 channel 交接、或去掉共享可变对象
- 正例 case：
  - `pr-1617`
  - `pr-1687`
  - `pr-1759`
  - `pr-2054`
  - `pr-30853`
  - `pr-37466`
  - `pr-9526`
- 边界 / 不应混入：
  - `pr-53245`
  - `pr-53887`
  - 这两条更像 `DDL callback 目标 job 过滤 / once 控制`

### 2. DDL hook callback 复用共享 context / store / txn

- 收敛状态：可优先起正式 JSON
- 确定性观察信号：
  - callback 沿用外部创建的 `context`、`store`、`txn` 或类似运行时对象
  - callback 和主线程/其他路径同时访问同一个对象，而不是每次独立创建
  - 修复通常是 callback 内自建上下文，或避免跨 callback 共享事务/上下文对象
- 正例 case：
  - `pr-1903`
  - `pr-2167`
- 边界 / 相邻：
  - 不要与“写共享变量无锁”合并
  - 不要与“复用共享 TestKit/session 执行 SQL”合并

### 3. DDL hook callback 复用共享 TestKit / session 执行 SQL

- 收敛状态：可优先起正式 JSON
- 确定性观察信号：
  - callback / goroutine 内直接复用外层 `TestKit`、`session`、`test helper`
  - 同一会话对象被 callback 和主测试线程并发用来执行 SQL / DDL
  - 修复通常是每个 goroutine 单独建 session / TestKit，或把额外 SQL 挪回主线程串行执行
- 正例 case：
  - `pr-13447`
  - `pr-1656`
  - `pr-1669`
- 边界 / 相邻：
  - 不要与“同一 session 的上一个 RecordSet 未 Close 就执行下一条 SQL”混在一起

### 4. goroutine 闭包复用外层 `err` 变量

- 收敛状态：可优先起正式 JSON
- 确定性观察信号：
  - `go func(...) { err = ... }()` 或等价写法
  - goroutine 内直接给外层 `err` 赋值
  - 主线程或其他 goroutine 会并发读这个 `err`
  - 修复通常是改成 goroutine 内局部变量 `err := ...`
- 正例 case：
  - `pr-672`
  - `pr-43704`
- 混合但相关的 case：
  - `pr-625`
  - `pr-27436`
  - 这两条还混有 `WaitGroup/lifecycle` 问题，不应直接当成纯净锚点

### 5. goroutine 闭包捕获循环变量

- 收敛状态：可优先起正式 JSON
- 确定性观察信号：
  - 在 `for range` / 普通循环里启动 goroutine
  - 闭包直接用循环变量 `i` / `store` / `sql` / `plan` / `case`，没有做重新绑定
  - 并发子测试或后台 goroutine 因此读到错误对象
  - 修复通常是把循环变量作为参数传入闭包，或在循环体内新建局部副本
- 正例 case：
  - `pr-14549`
  - `pr-23866`
  - `pr-33862`
  - `pr-36202`
- 混合但相关的 case：
  - `pr-31244`
  - 这条还混有 goroutine 未等待退出

### 6. BindHandle 单例 `sctx` 并发复用

- 收敛状态：可优先起正式 JSON
- 确定性观察信号：
  - 某个长期存活的 handle / manager 持有单例 `sctx`
  - 多条路径会并发进入这个 handle，并共用该 `sctx`
  - 修复通常是给关键区加锁，或避免把单例 session 上下文暴露给并发路径
- 正例 case：
  - `pr-49092`
  - `pr-49164`
- 边界 / 相邻：
  - 不要和 `BindHandle 共享 parser 并发使用` 合并

### 7. BindHandle 共享 parser 并发使用

- 收敛状态：单例候选，但代码信号很强
- 确定性观察信号：
  - `BindHandle` 或相似组件持有共享 parser 实例
  - 多个 goroutine / 并发更新路径直接复用这个 parser
  - 修复通常是每次新建 parser，或在共享 parser 外层加锁
- 正例 case：
  - `pr-10321`
- 备注：
  - 目前只有一个高纯度正例，但机制非常确定，可以先保留

### 8. 同一 session 的上一个 `RecordSet` 未 Close 就执行下一条 SQL

- 收敛状态：单例候选，但代码信号很强
- 确定性观察信号：
  - 同一 `session` / `TestKit` 返回的 `RecordSet` 尚未 `Close`
  - 随后立刻在同一会话上继续执行下一条 SQL
  - 前一个结果集的清理 / schema 变更 / 后台操作和后续 SQL 竞争
  - 修复通常是显式 `Close` 旧 `RecordSet`，或避免同一 session 重入
- 正例 case：
  - `pr-49348`
- 边界 / 相邻：
  - `pr-35134` 更像 `RecordSet close / lifecycle`
  - `pr-44438`
  - `pr-44783`
  - 这两条更像 `DDL callback 与主线程共享 RecordSet/jobID/err`

### 9. 测试改写全局 lease 干扰后台 refresh / loader

- 收敛状态：候选已成型，但还要继续补 case
- 确定性观察信号：
  - 测试直接改 `bindinfo.Lease` / stats lease / 同类全局 lease 开关
  - domain / loader / background worker 仍然存活
  - 断言会与后台 refresh / reload / drop / capture 路径竞争
  - 修复通常是把 lease 设为 `0` 或显式停掉后台 worker，并保证修改时机先于 bootstrap / loop 启动
- 正例 case：
  - `pr-66559`
- 混合但相关的 case：
  - `pr-7901`
  - 这条还混有 `SetSystemTZ` 一次性全局初始化问题

### 10. 多个 cop iterator 共享可变 `KV request / RequestBuilder`

- 收敛状态：单例候选，但代码信号很强
- 确定性观察信号：
  - 多个 iterator / worker 并发运行
  - 它们复用同一个 `RequestBuilder` 或可变 request 对象
  - 后续构建 / 发送请求时会互相覆盖状态
  - 修复通常是每个 iterator 独立构建 request
- 正例 case：
  - `pr-61376`

### 11. `WaitGroup.Add(1)` 晚于任务投递，导致 `Done()` 抢先发生

- 收敛状态：单例候选，但代码信号很强
- 确定性观察信号：
  - 任务先被送入 channel / 队列，worker 可以立刻执行
  - `WaitGroup.Add(1)` 却在投递之后才做
  - worker 可能先 `Done()`，把计数打成负数或造成收尾错乱
  - 修复通常是先 `Add(1)`，再投递任务
- 正例 case：
  - `pr-66703`
- 边界 / 不应混入：
  - `pr-42129` 更像 `WaitGroup Add/Wait` 的 shutdown sequencing

### 12. 并发 callback 共享结果 slice，`append/reset/snapshot` 未统一加锁

- 收敛状态：单例候选，但代码信号很强
- 确定性观察信号：
  - callback / hook 会把结果写入共享 slice
  - 主线程还会对同一个 slice 做 `reset`、读快照或断言
  - 缺少统一的 `mutex` / 所有权交接
  - 修复通常是所有 `append/reset/snapshot` 都放到同一把锁下
- 正例 case：
  - `pr-65523`

## 下一步待扩展方向

当前草案里列出的稳定候选已经全部落成正式 JSON。下一步如果继续扩展，应回到 `502` case 台账里继续寻找新的高纯度细簇，例如：

- 更细的 `manager 级共享 map 并发读写` 子模式
- `shared mock / processor-local state` 之外的其他 fresh-instance-per-call 子族
- `copy-on-write / atomic swap` 与 `publish-once` 之间需要继续拆分的共享结构子模式
- 更细的 `BindHandle / bind cache` 子族

## 暂不并入本轮主簇的边界

- `cancel / close / teardown` 生命周期问题
- `timing / sleep / Eventually / failpoint` 窗口问题
- `state-machine / owner-transfer / stale-state` 问题
- `exact-order / exact-count` brittle assertion
