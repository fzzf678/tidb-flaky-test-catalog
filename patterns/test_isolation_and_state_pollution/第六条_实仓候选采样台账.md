这份台账只记录一件事：

- 对第六条 sibling 当前 3 档 repo-scan 候选做一轮**人工 file-level triage**

它不是：

- verdict

这里的目标更窄：

- 看 `broad_recall` / `harness_teardown_weak_recall` / `package_sandbox_weak_recall` 在真实 TiDB HEAD 上，主要到底扫出来了什么
- 把这些命中文件先粗分成：
  - `likely_in_sibling`

  - `generic_scaffolding`

## Hard Rules

1. 这是 **file-level triage**，不是 patch-level verdict。
2. `likely_in_sibling` 只表示：
   - 这个文件当前暴露出的命中形态和第六条机制很接近
3. `likely_boundary` 只表示：
   - 这个文件更像相邻边界，例如：
     - global-knob / async boundary
     - background-component boundary
     - namespace boundary
4. `generic_scaffolding` 只表示：
   - 当前文件更像 routine `main_test.go` / bootstrap / teardown 模板
   - 不能单独拿它证明“这条候选没价值”
5. 如果后续真的要下 verdict，仍然要回到：
   - patch review
   - `subpatterns/server_harness_package_sandbox_级_test_global_状态必须重建或清理.json`

## 标签定义

| label | 含义 |
|---|---|
| `likely_in_sibling` | 当前文件已经直接出现 harness restart / shared harness state recreate / explicit server-harness teardown 语义 |
| `likely_boundary` | 当前文件的主导信号更像 global knob / schema lease / background stats / mixed async boundary |
| `generic_scaffolding` | 当前文件更像 package `TestMain`、普通 bootstrap/close 模板，只有弱相关线索 |

### 当前判断

- 这一桶当前大部分命中都只是 generic `main_test.go` scaffolding。
- 它的真正价值不是“默认第一跳”，而是：
  - 当你明确想追 package sandbox / package-level harness 分支时，再定向补召回。
- 当前还不适合把下面这些 generic scaffolding 词直接当 down-rank：
  - `goleak.VerifyTestMain(...)`
  - `config.UpdateGlobal(...)`
  - `testsetup.SetupForCommonTest(...)`
  - `tikv.EnableFailpoints()`
  - `autoid.SetStep(...)`
- 原因是：
  - 这些词既贡献噪声，也确实会出现在 package sandbox 分支的真实上下文里。
